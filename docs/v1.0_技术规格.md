# GameData Studio — Excel Add-in 技术规格文档

> **用途**：供 Claude Code 开发参考的完整技术规格  
> **版本**：v1.0 | 2026-03-10  
> **项目类型**：Office Web Add-in（侧边栏加载项）  
> **技术栈**：React + TypeScript + Office JS API  
> **目标**：完整替代现有 VBA 导出系统

---

## 一、项目概述

### 1.1 背景

RPG/MMO 游戏的数值策划团队（10人以上）使用 Excel 管理全部游戏数据配置。所有数据集中在一个 .xlsm 工作簿中，通过 OneDrive 同步实现多人协作（同时编辑一般不超过2-3人），使用 VBA 宏按版本和地区筛选数据并导出，Git 管理导出后的文件。

### 1.2 Add-in 定位

在 Excel 侧边栏中提供增强功能面板，完全保留 Excel 原生编辑能力，不改变工作簿的任何结构。策划继续在表格区域编辑数值，Add-in 负责导出、校验、可视化等增强操作。

### 1.3 v1.0 功能范围

| 功能 | 说明 |
|------|------|
| 导出引擎 | 完整移植 VBA 的版本筛选、差异对比、文件写入逻辑 |
| 错误码机制 | 导出过程中的错误精确报告（错误码+位置+说明） |
| 导出前校验 | 版本区间格式、数据区空值等基础校验 |
| Git 集成 | 拉取最新+提交推送，智能检测 Git 环境 |
| 网页协作导出 | OneDrive 同步方式，替代 VBA 的检测机制 |
| 基础侧边栏 | 导出按钮、Git 按钮、进度显示、结果列表、错误日志 |

## 二、Excel 工作簿结构

### 2.1 工作簿组成

工作簿包含三类工作表：

**元数据表（3张）**：
- `表格输出` — 控制面板，存放导出状态、版本信息、操作日志
- `配置设置表` — 版本模板、线路模板、人员代码、Git配置
- `表名对照` — 中英文表名映射、输出开关、版本区间

**数据表（N张）**：如物品表、装备表、配置表、关卡表等，存放实际游戏数据

**辅助表**：模板说明、说明表等，不参与导出

### 2.2 元数据表详细结构

#### 2.2.1 配置设置表

通过标记文字定位各配置区域：

**`#版本列表#` 区域**（版本模板定义）：
```
标记行:   #版本列表#  | 输出线路  | Git目录              | 备注目录
数据行1:  主干版本     | 1        |                      |
数据行2:  国内         | 2        | C:\...\ceshi\{0}     |
数据行3:  版号包       | 3        |                      |
数据行4:  日本         | 4        |                      |
数据行5:  韩国         | 5        |                      |
数据行6:  越南         | 6        |                      |
数据行7:  国际版本     | 7        |                      |
数据行8:  大陆小游戏版本| 8        |                      |
```

- 第1列：版本名称（唯一标识）
- 第2列：输出线路编号（对应线路列表中的ID，一对一关系）
- 第3列：Git目录模板，`{0}` 替换为版本号，`{1}` 替换为当前用户名

**`#线路列表#` 区域**（线路定义）：
```
标记行:   #线路列表#  | 线路字段   | 备注
数据行1:  1          | roads_0    | 总线路
数据行2:  2          | roads_1    | 简体中文线路
数据行3:  3          | roads_2    | 版号包线路
数据行4:  4          | roads_3    | 韩文线路
数据行5:  5          | roads_4    | 日文线路
数据行6:  6          | roads_5    | 越南线路
数据行7:  7          | roads_6    | 多语言线路
数据行8:  8          | roads_7    | 其他1线路
数据行9:  9          | roads_8    | 其他2线路
```

- 第1列：线路ID
- 第2列：线路字段名（对应数据表中的列名）
- **关键**：版本的"输出线路"编号 与 线路列表的"线路ID" 对应，如版本"国内"输出线路=2 → 线路ID=2 → 字段 `roads_1`

**`#人员代码#` 区域**：
```
标记行:   #人员代码#  | 人员名称  | 配置代码  | 机器码
数据行1:  1          | 孙青松    | s         | 211194084900315
```

- 配置代码用于版本号后缀（如 `2s` 表示成员 s 配置的2.0版本数据）
- 机器码用于识别操作机器

**`#Git通用提交日志#` 区域**：
```
标记行:   #Git通用提交日志#
数据行:   --story--user=通用数据表配置需求---
```

**`#配置开关#` 区域**：
```
标记行:   #配置开关#     | 参数
数据行:   自动弹出路径    | False
```

#### 2.2.2 表格输出（控制面板）

```
#工作状态#        | {状态文本}         ← 导出/上传过程中的实时状态
#数据表版本#      |                    ← 上一行偏移(−1, +1)位置存放版本序列号
                  | {版本号.序列号}     ← 如 7.5.1152
#输出版本#        | {版本名称}         ← 如 "国内"
#输出版本号#      | {版本号}           ← 如 7.5
#操作人#          | {人员名称}         ← 如 "孙青松"
#操作标识#        | {标识文本}         ← 如 "空|孙青松"

#输出表格结果#
  输出表格数      | {数量}

#输出表格列表#                         ← 标记下方逐行列出变更的文件路径
#输出错误列表#                         ← 标记下方逐行记录错误日志（时间|错误号|过程名|工作表|附加信息|错误描述）
```

#### 2.2.3 表名对照

```
#输出控制#  | 功能表名      | 输出表名          | 是否输出表
1          | 系统表        | SystemInfo        | True
1          | 配置表        | GameConfig        | True
1          | 物品表        | GameItem          | True
1          | 宝箱表        | GameBox           | True
1          | 玩家等级表    | PlayerLevel       | True
1          | 属性强化表    | AttributeEnhance  | True
1          | 装备表        | Equipment         | True
1          | 等级效果表    | LevelEffect       | True
1          | 等级消耗表    | LevelConsume      | True
1          | 关卡表        | GameStage         | True
1          | 语言表        | LanguageWord      | True
```

- 第1列（A列）：版本区间字符串，控制该表从哪个版本开始存在
- 第2列：中文表名（= Excel 工作表名）
- 第3列：英文表名（= 导出文件名）
- 第4列：是否输出（True/False 全局开关）

## 三、数据表布局规范

### 3.1 数据表的两种布局

每张数据表由三个区域组成，其中**版本列控制区**可选：

#### 布局A：有列版本控制（如物品表、系统表）

```
Row 1:  [空..] | 版本列属 | version_c | {列1版本值} | {列2版本值} | ...
Row 2~N: [空..] | {线路字段} | {线路字段}| {列1线路值} | {列2线路值} | ...  ← 可选的多行线路控制
Row ...: (可能有空行)
Row M:  version_r | roads_0 | roads_1 | ... | roads_8 | [空列..] | #配置区域# | {字段定义} | {字段定义} | ...
Row M+1: 版本行属 | 简体    | 简体净化 | ... | 其他2线路 | [空列..] |           | {中文描述} | {中文描述} | ...
Row M+2: {版本值} | {0/1}  | {0/1}   | ... | {0/1}   |          |           | {数据}     | {数据}     | ...
```

**version_c 区域详细说明**：
- "版本列属"文字标记所在列的右一列是 `version_c` 文字
- `version_c` 右侧依次对应数据区每一列的版本区间值
- version_c 下方可以有线路控制行（结构与 version_r 区域的线路列对称）
- 列版本控制区和 version_r 所在行之间可能有空行，间距不固定

**实际示例（系统表）**：
```
Row 1: [L]版本列属 [M]version_c [N]1 [O]1 [P]1 [Q]1 [R]1 [S]~1 [T]1 [U]1.3 [V]1
                                                              ↑ ~1=1.0前   ↑ 1.3起
Row 6: [A]version_r [B]roads_0 [C]roads_1 ... [M]#配置区域# [N]id=int [O]name=string ...
```

#### 布局B：无列版本控制（如装备表、配置表、语言表）

```
Row 1:  version_r | roads_0 | roads_1 | ... | roads_8 | [空列..] | #配置区域# | {字段定义} | ...
Row 2:  版本行属  | 简体    | 简体净化 | ... | 其他2线路 |          |           | {中文描述} | ...
Row 3:  {版本值}  | {0/1}  | {0/1}   | ... | {0/1}   |          |           | {数据}     | ...
```

### 3.2 关键定位规则

**全部通过文字查找定位，不依赖固定行列号**：

| 标记文字 | 定位目标 | 查找范围 |
|---------|---------|---------|
| `#配置区域#` | 数据区起点。右侧第一列就是第一个数据列 | 整个工作表 |
| `version_r` | 行版本控制区起点。所在行即为字段定义行（表头第一行），下一行为中文描述行，之后为数据行 | A列 |
| `version_c` | 列版本控制区起点。右侧依次为各数据列的版本值 | 整行扫描 |
| `版本列属` | version_c 区域的标记辅助文字 | 辅助定位用 |
| `版本行属` | version_r 下方中文描述行的A列标记 | 辅助定位用 |

### 3.3 字段定义行格式

`#配置区域#` 所在行的右侧列，格式为 `{字段名}={类型}`：

```
id=int | name=string | rewards=int[] | monsterList=int[][] | desc=string
```

支持的类型：`int`, `string`, `int[]`, `int[][]`, `float` 等  
部分字段名有前缀 `key_`（如 `key_序号id`），表示该列是主键  
部分字段名有前缀 `language_`（如 `language_物品名称`），表示该字段需要多语言处理

### 3.4 中文描述行

`版本行属` 所在行，在数据列区域包含每个字段的中文描述：

```
物品id | language_物品名称 | language_物品描述 | 物品图标 | 物品品质 | ...
```

### 3.5 `#配置区域#` 位置不固定

`#配置区域#` 可能出现在不同的列（M列、N列等），取决于版本控制区和数据区之间的空列数量。必须通过查找定位。

## 四、核心导出逻辑

### 4.1 导出主流程

```
1. 加载配置
   ├── 从「配置设置表」加载：版本模板、线路模板、人员代码、Git配置、开关
   ├── 从「表格输出」加载：当前输出版本名、版本号、版本序列号
   └── 从「表名对照」加载：待处理表格列表（中英文名、输出开关、版本区间）
   → 任何配置加载失败：错误码 1xxx，中断导出

2. 确定筛选参数
   ├── 目标版本号 = 输出版本号（如 7.5）
   ├── 查找输出版本名对应的线路编号（如 "国内" → 2）
   └── 映射线路编号到字段名（如 2 → roads_1）

3. Git 环境检测（详见第八章）
   ├── 检查系统是否安装 Git
   ├── 检查输出目录是否在 Git 仓库内
   └── 不可用 → 记录 warning（错误码 4002/4006），跳过 Git 操作，继续导出

4. 加载所有源数据到内存
   └── 遍历表名对照中的每张表，检查版本区间是否包含目标版本号
       ├── 是 → 读取该工作表的全部相关数据到内存
       └── 找不到工作表 → 错误码 2001（warning），跳过该表

5. 导出前校验（详见第六章）
   ├── 版本区间格式校验（错误码 2101-2104）
   ├── 数据区空值检查（错误码 2201）
   └── 校验结果作为 warning 记录，不中断导出

6. Git 拉取最新（仅 Git 可用时执行）
   └── git reset --hard && git clean -dfq && git pull
   → 拉取失败：错误码 4004（warning），继续导出

7. 创建/验证输出目录
   └── 根据版本模板的Git目录模板计算路径
   → 目录不可用：错误码 3001-3003，中断导出

8. 打开/创建「全部表.xlsx」
   └── 位于输出目录下
   → 失败：错误码 3010，中断导出

9. 逐表处理（核心）
   ├── 行版本筛选（详见 4.2）
   ├── 列版本筛选（详见 4.3）
   ├── 重复Key处理（详见 4.4）
   ├── 差异对比（详见第七章）
   └── 文件写入
   → 某表处理失败：错误码 2xxx/3xxx（warning），跳过该表继续

10. 收尾
    ├── 保存关闭「全部表.xlsx」
    ├── 版本序列号自增并写回控制表
    └── 更新「表格输出」的状态区域（结果列表+错误列表）
    └── 侧边栏显示完整的导出结果和校验警告
```

### 4.2 行版本筛选

对每一数据行（`version_r` 所在行之后的第2行起），执行以下检查：

```
输入：
  - targetVersion: 目标版本号（如 7.5）
  - primaryRoad: 版本对应的线路字段名（如 "roads_1"）
  - 当前行在版本控制区的数据：A列=版本区间, B列=roads_0值, C列=roads_1值, ...

判定流程：
  1. 读取 A 列的版本区间字符串
  2. 检查 targetVersion 是否在该区间内（调用 IsVersionInRange）
     → 不在区间内 → 排除该行
  3. 检查 roads_0 列的值：
     - 值为 "1" → 通过
     - 否则，将值作为版本区间字符串，调用 IsVersionInRange 检查
     → 不通过 → 排除该行
  4. 检查对应地区的 roads_X 列的值（同 roads_0 的逻辑）
     → 不通过 → 排除该行
  5. 三个条件全部满足 → 保留该行

关系：AND（三个条件必须同时满足）
```

**表头行（字段定义行和中文描述行）始终保留，不参与版本筛选。**

### 4.3 列版本筛选

仅当工作表中存在 `version_c` 标记时执行：

```
输入：
  - version_c 区域的数据（可能有多行：第1行=版本区间值, 后续行=线路控制值）
  - targetVersion, primaryRoad（同行筛选）

判定流程（对每一数据列）：
  1. 读取 version_c 第1行中该列对应的版本区间值
  2. 检查 targetVersion 是否在该区间内
     → 不在区间内 → 排除该列
  3. 如果 version_c 有多行（线路控制行存在）：
     - 检查 roads_0 对应的值
     - 检查 roads_X 对应的值
     → 逻辑与行筛选对称
  4. 全部条件满足 → 保留该列
```

### 4.4 版本区间解析（IsVersionInRange）

```
输入：versionRangeString（版本区间字符串）, targetVersion（目标版本号）
输出：Boolean

解析规则：
  ""（空值）     → 解析为 [0, 0.1)       → 几乎不导出
  "1.0"          → 解析为 [1.0, 99)      → 1.0及以上永久生效
  "1.0~2.5"      → 解析为 [1.0, 2.5)     → 左闭右开
  "~2.5"         → 解析为 [0, 2.5)       → 2.5以下
  "1.0~"         → 解析为 [1.0, 99)      → 同 "1.0"
  "1"            → 解析为 [1, 99)        → 值为 "1" 时视为版本区间（不是布尔开关）
  "3.5a"         → 提取数字部分 3.5      → 字母后缀（成员配置代码）被忽略
  "2s"           → 提取数字部分 2.0      → 成员代码 s 被忽略
  "1.2s~1.5s"    → 提取数字部分 [1.2, 1.5)

判定：targetVersion >= minVersion AND targetVersion < maxVersion

特殊情况：
  - roads_X 列的值为 "1" 时，含义是"在所有版本中启用"（解析为 [1, 99)）
  - roads_X 列的值为 "0" 或空时，该行/列在此线路下被排除
```

### 4.5 重复Key处理

```
筛选完成后，遍历结果数据（从第3行起，跳过两行表头）：
  - 第一数据列（#配置区域# 右侧第一列）作为 Key
  - 如果当前行的 Key 与前一行相同 → 用当前行覆盖前一行
  - 这是"简写模式"的核心机制：同一 Key 在不同版本区间有不同配置，
    筛选后保留靠后的（版本更高的）那行

示例（配置表导出版本 7.5）：
  原始数据：
    Row5: version_r=1~1.1  | id=3 | param=SUMMONS_WEAPON... | value=旧值
    Row6: version_r=1.2    | id=3 | param=SUMMONS_WEAPON... | value=新值
  筛选后（7.5 > 1.2 > 1.1，两行都保留）：
    → id=3 出现两次，保留后面那行（value=新值）
```

### 4.6 差异对比与文件写入

```
对比逻辑：
  1. 打开输出目录下的「全部表.xlsx」
  2. 查找英文名对应的工作表
     - 不存在 → 判定为新增，需要输出
     - 存在 → 读取旧数据，与筛选后的数据逐单元格对比
       - 行列数不同 → 判定为变更
       - 逐格 CStr 比较，任何不同 → 判定为变更
  3. 特殊：英文名为 "GameConfig" 的表总是判定为变更

输出操作（仅变更的表执行）：
  1. 写入独立 .xlsx 文件
     - 路径：{输出目录}\{英文名}.xlsx
     - 内容：筛选后的数据（含两行表头），格式设为文本（NumberFormat = "@"）
     - GameConfig 特殊处理：将第3行第3列的值替换为 "{版本号}.{序列号}"
  2. 更新「全部表.xlsx」
     - 查找或创建对应英文名的工作表
     - 清空旧数据，写入筛选后的数据
```

### 4.7 导出内容的精确定义

```
导出内容 = #配置区域# 标记右侧的所有列

导出行结构（自上而下）：
  第1行：字段定义行（version_r 所在行的数据列部分）  ← 如 id=int | name=string | ...
  第2行：中文描述行（version_r 下一行的数据列部分）  ← 如 物品id | 物品名称 | ...
  第3行起：筛选后的数据行

注意：两行表头的顺序跟原表一致，不做调换。
大部分表是字段定义在前、中文描述在后，但不强制。
```

## 五、错误码机制

### 5.1 错误码定义

导出过程中任何异常都必须给出：错误码 + 出错位置 + 具体描述。策划在侧边栏看到错误后能直接定位到问题。

```typescript
interface ExportError {
  code: number;              // 错误码
  severity: 'error' | 'warning';  // error=导出中断, warning=继续但记录
  tableName: string;         // 出错的表名
  location?: {               // 出错的具体位置（可选）
    sheetName: string;
    row: number;             // 1-indexed
    column: number;          // 1-indexed
    cellValue?: string;      // 出错单元格的值
  };
  message: string;           // 人类可读的错误描述
  procedure: string;         // 出错的函数/流程名
}
```

### 5.2 错误码表

```
1000-1099: 配置加载错误
  1001  找不到配置设置表
  1002  找不到控制表（表格输出）
  1003  找不到表名对照表
  1010  版本序列号非数字
```

### 5.3 错误处理流程

```
导出过程中的错误分为两类：

中断性错误（error）：
  - 配置加载失败（1000系列）→ 整个导出中断
  - 输出目录不可用（3001-3003）→ 整个导出中断
  - 全部表打开/保存失败（3010-3011）→ 整个导出中断

非中断性错误（warning）：
  - 某张表加载失败（2001）→ 跳过该表，继续处理其他表
  - 某张表筛选后为空（2202）→ 跳过该表，继续
  - 某张表文件写入失败（3101-3103）→ 记录错误，继续处理其他表
  - 数据区空值（2201）→ 记录位置，继续导出

所有错误都写入侧边栏的错误日志面板，同时写入「表格输出」的 #输出错误列表# 区域（双写）。
错误日志包含：时间、错误码、表名、位置（行列号）、错误描述。
策划可以点击错误条目跳转到出错的单元格。
```

### 5.4 ErrorHandler 实现

```typescript
class ErrorHandler {
  private errors: ExportError[] = [];

  /**
   * 记录错误
   * 同时写入内存列表和「表格输出」工作表
   */
  async logError(error: ExportError): Promise<void> {
    this.errors.push(error);

    // 写入「表格输出」的 #输出错误列表# 区域
    await Excel.run(async (context) => {
      const sheet = context.workbook.worksheets.getItem('表格输出');
      const marker = await excelHelper.findMarker(sheet, '#输出错误列表#');
      if (!marker) return;

      // 找到下一个空行
      const nextRow = this.findNextEmptyRow(marker);
      nextRow.getResizedRange(0, 5).values = [[
        new Date().toLocaleString(),       // 时间
        error.code,                         // 错误码
        error.procedure,                    // 过程名
        error.tableName,                    // 表名
        error.location ? `行${error.location.row} 列${error.location.column}` : '', // 位置
        error.message,                      // 描述
      ]];

      // 错误行标红
      if (error.severity === 'error') {
        nextRow.getResizedRange(0, 5).format.fill.color = '#FFE6E6';
      }
      await context.sync();
    });
  }

  /**
   * 获取所有错误（供侧边栏展示）
   */
  getErrors(): ExportError[] {
    return this.errors;
  }

  /**
   * 是否有中断性错误
   */
  hasCriticalErrors(): boolean {
    return this.errors.some(e => e.severity === 'error');
  }

  /**
   * 清空错误列表（导出开始前调用）
   */
  clear(): void {
    this.errors = [];
  }
}
```

## 六、导出前校验

### 6.1 校验时机

在导出流程的第 2 步（加载源数据到内存）之后、第 6 步（逐表处理）之前，对所有待导出的表执行一轮基础校验。校验不通过的项作为 warning 记录，不中断导出流程（策划可以在结果中查看）。

### 6.2 校验项

```
对每张待导出的数据表执行以下检查：

1. 版本区间格式校验（A列）
   遍历 version_r 以下的所有数据行的 A 列：
   - 检查是否使用了横线 - 而非波浪号 ~  → 错误码 2101
   - 尝试解析版本区间，格式无效  → 错误码 2102
   - 最小值 > 最大值  → 错误码 2103
   
2. 数据区空值检查

   - 发现空单元格  → 错误码 2201（warning，记录位置但不中断）
   
3. 结构完整性检查
   - 找不到 #配置区域# 标记  → 错误码 2010
   - 找不到 version_r 标记  → 错误码 2011
   - 数据区域无列  → 错误码 2012
```

## 七、差异对比详细逻辑

### 7.1 对比流程

```
对每张筛选完成的数据表，与「全部表.xlsx」中的对应工作表进行对比：

步骤1: 查找目标工作表
  - 在「全部表.xlsx」中查找英文名对应的工作表
  - 不存在 → 判定为「新增表」，需要输出
  - 存在 → 进入对比

步骤2: 维度对比
  - 读取旧数据的行数和列数
  - 与筛选后的行数和列数对比
  - 不同 → 判定为「变更」

步骤3: 逐单元格对比（维度相同时）
  - 将两个二维数组的每个元素转为字符串后对比
  - 任何一个不同 → 判定为「变更」
  - 全部相同 → 判定为「无变化」，跳过输出

步骤4: GameConfig 特殊处理
  - 英文名为 "GameConfig" 的表，总是判定为「变更」
  - 因为需要注入版本号，即使其他数据没变，版本号也不同

步骤5: 输出操作（仅「新增」和「变更」的表执行）
  - 写入独立 .xlsx 文件
  - 更新「全部表.xlsx」中的对应工作表
```

### 7.2 导出结果列表

```
导出完成后，变更文件列表写入两个位置：

1. 侧边栏的结果面板（实时显示）
2.「表格输出」的 #输出表格列表# 区域（双写，兼容 VBA 的 Git 上传逻辑）

列表内容：
  - 每个变更表的独立 .xlsx 文件完整路径
  - 最后一项：全部表.xlsx 的完整路径
  - 字体颜色标红（与 VBA 保持一致，用于 Git 上传时识别待提交文件）

同时更新 #输出表格结果# 区域的变更表数量。
```

## 八、Git 操作详细逻辑

### 8.1 Git 环境检测

```
在导出流程开始前，检测 Git 环境：

步骤1: 检查系统是否安装 Git
  - 尝试执行 git --version
  - 成功 → Git 可用
  - 失败 → 错误码 4006「系统未安装 Git」→ 跳过所有 Git 操作，仅执行导出
  - 侧边栏显示：「⚠ 未检测到 Git，导出正常执行但跳过 Git 拉取和推送」

步骤2: 检查输出目录是否在 Git 仓库内
  - 从输出目录开始，逐级向上查找 .git 文件夹
  - 找到 → 记录仓库根路径
  - 未找到 → 错误码 4002「输出目录不在 Git 仓库内」→ 跳过 Git 操作
  - 侧边栏显示：「⚠ 输出目录不在 Git 仓库内，跳过 Git 操作」

Git 环境检测的结果决定后续行为：
  - Git 可用 + 仓库存在 → 执行完整流程（拉取→导出→推送可用）
  - Git 不可用 或 仓库不存在 → 仅执行导出，Git 按钮置灰并显示原因
```

### 8.2 导出前 Git 拉取

```
目的：确保本地文件是最新的，避免导出后覆盖别人的更新。

执行时机：导出流程的第 3 步（在加载数据到内存之后，写入文件之前）

操作序列：
  1. git reset --hard      ← 丢弃所有本地未提交的修改
  2. git clean -dfq         ← 删除所有未跟踪的文件
  3. git pull               ← 拉取远程最新

命令拼接：
  {盘符}: && cd "{仓库根路径}" && git reset --hard && git clean -dfq && git pull

⚠ 关键行为：每次导出前会丢弃输出目录中的所有本地修改。
  这是设计意图——输出目录应该只包含由工具导出的文件，
  不应该有手动修改。如果需要手动修改导出文件，应该先提交再导出。

错误处理：
  - 拉取失败 → 错误码 4004（warning，不中断导出）
  - 侧边栏显示：「⚠ Git 拉取失败，将基于当前本地文件导出」
  - 策划可以选择继续导出或中止
```

### 8.3 Git 提交推送

```
执行时机：导出完成后，策划在侧边栏点击「Git 推送」按钮

前置条件：
  - 导出结果中有变更文件（modifiedFiles.length > 0）
  - Git 环境可用

操作序列：
  1. 逐个 git add 每个变更文件的完整路径
  2. git add 全部表.xlsx 的完整路径
  3. git commit -m "{提交信息}"
  4. git push

提交信息生成：
  模板从配置设置表 #Git通用提交日志# 读取
  支持占位符：{VerName} {VerNum} {SeqNum} {User} {Date}
  拼接方式：模板内容 + 版本名 + " " + 版本号 + "." + 序列号

待提交文件列表 = 导出结果中的 modifiedFiles（变更的 .xlsx + 全部表.xlsx）

错误处理：
  - push 被拒绝（远程有更新）→ 提示「远程仓库有更新，请先拉取再推送」
  - 其他失败 → 错误码 4003/4005，显示 Git 命令行输出帮助策划理解问题
```

### 8.4 Git 仓库路径查找

```typescript
class GitHandler {
  /**
   * 从输出目录向上查找 .git 文件夹
   */
  findRepoRoot(outputDirectory: string): string | null {
    let current = outputDirectory;
    while (current.length > 3) {
      if (fs.existsSync(path.join(current, '.git'))) {
        return current;
      }
      const parent = path.dirname(current);
      if (parent === current) break;
      current = parent;
    }
    return null;
  }

  /**
   * Git 环境综合检测
   */
  async checkEnvironment(outputDirectory: string): Promise<GitEnvironment> {
    const gitInstalled = await this.isGitInstalled();
    if (!gitInstalled) {
      return { available: false, reason: '系统未安装 Git', errorCode: 4006 };
    }
    const repoRoot = this.findRepoRoot(outputDirectory);
    if (!repoRoot) {
      return { available: false, reason: '输出目录不在 Git 仓库内', errorCode: 4002 };
    }
    return { available: true, repoRoot };
  }
}

interface GitEnvironment {
  available: boolean;
  repoRoot?: string;
  reason?: string;
  errorCode?: number;
}
```

## 九、配置设置表中的版本-线路映射详解

```
版本选择流程：

  1. 用户在「表格输出」的 #输出版本# 位置选择版本名（如 "国内"）
  2. 在「配置设置表」的 #版本列表# 中查找 "国内"
     → 获取输出线路编号 = 2
     → 获取 Git 目录模板 = C:\...\{0}
  3. 在「配置设置表」的 #线路列表# 中查找线路 ID = 2
     → 获取线路字段名 = roads_1
  4. 导出时对每行数据检查：
     - roads_0（总线路，固定检查）—— AND
     - roads_1（国内线路）—— 都通过才保留

简化规则：一个版本 = 一条线路 = 一个地区
  主干版本 → roads_0（总线路）
  国内     → roads_0 + roads_1
  版号包   → roads_0 + roads_2
  日本     → roads_0 + roads_4（注意不是 roads_3）
  韩国     → roads_0 + roads_4（→实际应为 roads_3 + roads_0，需按配置表实际数据）
  ...

⚠ roads_0 是总开关，所有版本导出时都会检查 roads_0。
  如果某行 roads_0 = 0，则该行在所有版本中都不会导出。
```

## 十三、VBA 已知 Bug 及 Add-in 修正建议

在分析 VBA 代码过程中发现以下问题，Add-in 实现时应修正：

1. **clsDataTable.ApplyFilters 列筛选中的变量名错误**  
   VBA 代码第二遍列筛选中，`includeRow = False` 应为 `includeCol = False`。  
   → Add-in 中使用独立的 `includeCol` 变量。

2. **modErrorHandler.InitializeFallbackErrorLog 中错误使用 New**  
   `Dim modUtilities As modUtilities` 然后 `Set modUtilities = New modUtilities`，标准模块不能 New。  
   → Add-in 中 utils 模块使用纯函数导出，不存在此问题。

3. **LogToFile 默认禁用**  
   `LogToFile` 函数入口直接 `Exit Sub`，调试日志实际不工作。  
   → Add-in 中实现可开关的 console + 文件双写日志。

## 十四、验证策略

### P0 阶段验证标准

Add-in 的导出结果必须与 VBA 导出结果 100% 一致。验证方法：

1. **单元测试**：覆盖 VersionFilter 的所有边界条件
   - 空值、纯数字、带字母后缀、带~范围、省略最小值、省略最大值
   - 值为 "1"、"0"、空的 roads 列

2. **集成测试**：使用实际模板数据
   - 对每张表执行筛选，与 VBA 导出结果逐行逐列对比
   - 重点验证：有版本差异的行（如物品表 Row32-53）、有列版本控制的表（如系统表）、配置表的重复Key处理

3. **回归测试**：切换不同版本导出
   - 版本 1.0 导出（应排除 1.2+、1.5+ 等行）
   - 版本 7.5 导出（应包含所有行）
   - 验证 GameConfig 的版本号注入

## 十五、后续需要确认的事项

以下事项在开发过程中可能需要进一步确认：

1. **Office JS API 的文件系统访问能力**：桌面版 Add-in 能否直接写入本地文件？如果不能，需要考虑替代方案（如通过 Node.js 中间层或让用户手动保存）。

2. **大表的性能表现**：Office JS 的 `Range.getValues()` 对 100+ 行 × 20+ 列的表性能如何？可能需要分批读取策略。

3. **version_c 多行线路控制的实际数据样本**：当前模板中 version_c 只有一行，需要获取有多行线路控制的实际示例来验证列筛选逻辑。

4. **Git 命令行执行方式**：Office JS Add-in 中如何执行系统命令（git）？可能需要通过 Node.js 进程或其他方式。
