# GameData Studio — Excel Add-in 技术规格文档

> **用途**：供 Claude Code 开发参考的完整技术规格
> **版本**：v1.0 | 2026-03-11（与代码同步更新）
> **项目类型**：Office Web Add-in（侧边栏加载项）
> **技术栈**：React + TypeScript + Office JS API + ExcelJS
> **目标**：完整替代现有 VBA 导出系统

---

## 一、项目概述

### 1.1 背景

RPG/MMO 游戏的数值策划团队（10人以上）使用 Excel 管理全部游戏数据配置。所有数据集中在一个 .xlsm 工作簿中，通过 OneDrive 同步实现多人协作（同时编辑一般不超过2-3人），使用 VBA 宏按版本和地区筛选数据并导出，Git 管理导出后的文件。

### 1.2 Add-in 定位

在 Excel 侧边栏中提供增强功能面板，完全保留 Excel 原生编辑能力，不改变工作簿的任何结构。策划继续在表格区域编辑数值，Add-in 负责导出、校验、可视化等增强操作。

### 1.3 v1.0 功能范围

| 功能 | 说明 | 状态 |
|------|------|------|
| 导出引擎 | 完整移植 VBA 的版本筛选、差异对比、文件写入逻辑 | 已完成 |
| 错误码机制 | 导出过程中的错误精确报告（错误码+位置+说明） | 已完成 |
| 导出前校验 | 版本区间格式、Excel错误值、空格检测等校验 | 已完成 |
| Git 集成 | 生成 Git 推送命令并复制到剪贴板，供用户手动执行 | 已完成 |
| 网页协作导出 | OneDrive 同步方式，替代 VBA 的检测机制 | v1.0 暂未实现，后续考虑 |
| 侧边栏 UI | 蓝色header + 4个tab页签（导出/预览/校验/Git）+ 内联结果展示 | 已完成 |
| 文件系统 | 通过本地服务 /api/write-file 和 /api/read-file 访问磁盘 | 已完成 |

## 二、Excel 工作簿结构

### 2.1 工作簿组成

工作簿包含三类工作表：

**元数据表（3张）**：
- `表格输出` — 控制面板，存放导出状态、版本信息、操作日志
- `配置设置表` — 版本模板、线路模板、人员代码、Git配置
- `表名对照` — 中英文表名映射、输出开关、版本区间

**数据表（N张）**：如物品表、装备表、配置表、关卡表等，存放实际游戏数据

**辅助表**：模板说明、说明表等，不参与导出

### 2.2 元数据表详细结构

#### 2.2.1 配置设置表

通过标记文字定位各配置区域：

**`#版本列表#` 区域**（版本模板定义）：
```
标记行:   #版本列表#  | 输出线路  | Git目录              | 备注目录
数据行1:  主干版本     | 1        |                      |
数据行2:  国内         | 2        | C:\...\ceshi\{0}     |
数据行3:  版号包       | 3        |                      |
数据行4:  日本         | 4        |                      |
数据行5:  韩国         | 5        |                      |
数据行6:  越南         | 6        |                      |
数据行7:  国际版本     | 7        |                      |
数据行8:  大陆小游戏版本| 8        |                      |
```

- 第1列：版本名称（唯一标识）
- 第2列：输出线路编号（对应线路列表中的ID，一对一关系）
- 第3列：Git目录模板，`{0}` 替换为版本号，`{1}` 替换为当前用户名

**`#线路列表#` 区域**（线路定义）：
```
标记行:   #线路列表#  | 线路字段   | 备注
数据行1:  1          | roads_0    | 总线路
数据行2:  2          | roads_1    | 简体中文线路
数据行3:  3          | roads_2    | 版号包线路
数据行4:  4          | roads_3    | 韩文线路
数据行5:  5          | roads_4    | 日文线路
数据行6:  6          | roads_5    | 越南线路
数据行7:  7          | roads_6    | 多语言线路
数据行8:  8          | roads_7    | 其他1线路
数据行9:  9          | roads_8    | 其他2线路
```

- 第1列：线路ID
- 第2列：线路字段名（对应数据表中的列名）
- **关键**：版本的"输出线路"编号 与 线路列表的"线路ID" 对应，如版本"国内"输出线路=2 → 线路ID=2 → 字段 `roads_1`

**`#人员代码#` 区域**：
```
标记行:   #人员代码#  | 人员名称  | 配置代码  | 机器码
数据行1:  1          | 孙青松    | s         | 211194084900315
```

- 配置代码用于版本号后缀（如 `2s` 表示成员 s 配置的2.0版本数据）
- 机器码用于识别操作机器

**`#Git通用提交日志#` 区域**：
```
标记行:   #Git通用提交日志#
数据行:   --story--user=通用数据表配置需求---
```

**`#配置开关#` 区域**：
```
标记行:   #配置开关#     | 参数
数据行:   自动弹出路径    | False
```

#### 2.2.2 表格输出（控制面板）

```
#工作状态#        | {状态文本}         ← 导出/上传过程中的实时状态
#数据表版本#      |                    ← 上一行偏移(−1, +1)位置存放版本序列号
                  | {版本号.序列号}     ← 如 7.5.1152
#输出版本#        | {版本名称}         ← 如 "国内"
#输出版本号#      | {版本号}           ← 如 7.5
#操作人#          | {人员名称}         ← 如 "孙青松"
#操作标识#        | {标识文本}         ← 如 "空|孙青松"

#输出表格结果#
  输出表格数      | {数量}

#输出表格列表#                         ← 标记下方逐行列出变更的文件路径
#输出错误列表#                         ← 标记下方逐行记录错误日志（时间|错误号|过程名|工作表|附加信息|错误描述）
```

#### 2.2.3 表名对照

```
#输出控制#  | 功能表名      | 输出表名          | 是否输出表
1          | 系统表        | SystemInfo        | True
1          | 配置表        | GameConfig        | True
1          | 物品表        | GameItem          | True
1          | 宝箱表        | GameBox           | True
1          | 玩家等级表    | PlayerLevel       | True
1          | 属性强化表    | AttributeEnhance  | True
1          | 装备表        | Equipment         | True
1          | 等级效果表    | LevelEffect       | True
1          | 等级消耗表    | LevelConsume      | True
1          | 关卡表        | GameStage         | True
1          | 语言表        | LanguageWord      | True
```

- 第1列（A列）：版本区间字符串，控制该表从哪个版本开始存在
- 第2列：中文表名（= Excel 工作表名）
- 第3列：英文表名（= 导出文件名）
- 第4列：是否输出（True/False 全局开关）

## 三、数据表布局规范

### 3.1 数据表的两种布局

每张数据表由三个区域组成，其中**版本列控制区**可选：

#### 布局A：有列版本控制（如物品表、系统表）

```
Row 1:  [空..] | 版本列属 | version_c | {列1版本值} | {列2版本值} | ...
Row 2~N: [空..] | {线路字段} | {线路字段}| {列1线路值} | {列2线路值} | ...  ← 可选的多行线路控制
Row ...: (可能有空行)
Row M:  version_r | roads_0 | roads_1 | ... | roads_8 | [空列..] | #配置区域# | {字段定义} | {字段定义} | ...
Row M+1: 版本行属 | 简体    | 简体净化 | ... | 其他2线路 | [空列..] |           | {中文描述} | {中文描述} | ...
Row M+2: {版本值} | {0/1}  | {0/1}   | ... | {0/1}   |          |           | {数据}     | {数据}     | ...
```

**version_c 区域详细说明**：
- "版本列属"文字标记所在列的右一列是 `version_c` 文字
- `version_c` 右侧依次对应数据区每一列的版本区间值
- version_c 下方可以有线路控制行（结构与 version_r 区域的线路列对称）
- 列版本控制区和 version_r 所在行之间可能有空行，间距不固定

**实际示例（系统表）**：
```
Row 1: [L]版本列属 [M]version_c [N]1 [O]1 [P]1 [Q]1 [R]1 [S]~1 [T]1 [U]1.3 [V]1
                                                              ↑ ~1=1.0前   ↑ 1.3起
Row 6: [A]version_r [B]roads_0 [C]roads_1 ... [M]#配置区域# [N]id=int [O]name=string ...
```

#### 布局B：无列版本控制（如装备表、配置表、语言表）

```
Row 1:  version_r | roads_0 | roads_1 | ... | roads_8 | [空列..] | #配置区域# | {字段定义} | ...
Row 2:  版本行属  | 简体    | 简体净化 | ... | 其他2线路 |          |           | {中文描述} | ...
Row 3:  {版本值}  | {0/1}  | {0/1}   | ... | {0/1}   |          |           | {数据}     | ...
```

### 3.2 关键定位规则

**全部通过文字查找定位，不依赖固定行列号**：

| 标记文字 | 定位目标 | 查找范围 |
|---------|---------|---------|
| `#配置区域#` | 数据区起点。右侧第一列就是第一个数据列 | 整个工作表 |
| `version_r` | 行版本控制区起点。所在行即为字段定义行（表头第一行），下一行为中文描述行，之后为数据行 | A列 |
| `version_c` | 列版本控制区起点。右侧依次为各数据列的版本值 | 整行扫描 |
| `版本列属` | version_c 区域的标记辅助文字 | 辅助定位用 |
| `版本行属` | version_r 下方中文描述行的A列标记 | 辅助定位用 |

### 3.3 字段定义行格式

`#配置区域#` 所在行的右侧列，格式为 `{字段名}={类型}`：

```
id=int | name=string | rewards=int[] | monsterList=int[][] | desc=string
```

支持的类型：`int`, `string`, `int[]`, `int[][]`, `float` 等  
部分字段名有前缀 `key_`（如 `key_序号id`），表示该列是主键  
部分字段名有前缀 `language_`（如 `language_物品名称`），表示该字段需要多语言处理

### 3.4 中文描述行

`版本行属` 所在行，在数据列区域包含每个字段的中文描述：

```
物品id | language_物品名称 | language_物品描述 | 物品图标 | 物品品质 | ...
```

### 3.5 `#配置区域#` 位置不固定

`#配置区域#` 可能出现在不同的列（M列、N列等），取决于版本控制区和数据区之间的空列数量。必须通过查找定位。

## 四、核心导出逻辑

### 4.1 导出主流程

```
1. 加载配置（步骤1）
   ├── 一次性读取三张元数据表的 UsedRange 到内存（SheetSnapshot）
   ├── 从「配置设置表」解析：版本模板、线路模板、人员代码、Git配置、开关
   ├── 从「表格输出」解析：当前输出版本名、版本号、版本序列号
   └── 从「表名对照」解析：待处理表格列表（中英文名、输出开关、版本区间）
   → 任何配置加载失败：错误码 1xxx，中断导出

2. 初始化筛选器（步骤2）
   ├── 目标版本号 = 输出版本号（如 7.5）
   ├── 查找输出版本名对应的线路编号（如 "国内" → 2）
   └── 映射线路编号到字段名（如 2 → roads_1）
   → 无多线路支持：一个版本固定对应一条线路

3. 加载所有源数据到内存（步骤3）
   └── 遍历表名对照中的每张表
       ├── 检查输出开关和版本区间
       ├── 读取工作表 UsedRange → SheetSnapshot
       ├── 解析标记（version_r / #配置区域# / version_c）
       ├── 找不到 version_r → 返回 null，该表不导出
       └── 找不到工作表 → 错误码 2001（warning），跳过该表

4. 导出前校验（步骤4，详见第六章）
   ├── 配置区域（versionRowData）：允许空值，只检测 Excel 错误值和版本格式
   ├── 列版本区域（versionColData）：同上
   ├── 主数据区：检测 Ctrl+A 有效区域内的 Excel 错误值和纯空格
   └── 校验结果作为 warning 记录，不中断导出

5. 加载已有「全部表.xlsx」用于差异对比（步骤5）
   └── 通过本地服务 /api/read-file 读取输出目录下的文件
   → 不存在或读取失败：创建空工作簿，所有表均视为新增

6-8. 逐表处理（步骤6-8，核心）
   ├── 行版本筛选（详见 4.2）
   ├── 列版本筛选（详见 4.3）
   ├── 重复Key处理（详见 4.4）
   ├── 差异对比（详见第七章）
   ├── 生成独立 .xlsx 文件（ExcelJS 生成 ArrayBuffer）
   └── 通过本地服务 /api/write-file 写入磁盘
   → 某表处理失败：错误码 3101（warning），跳过该表继续

9. 保存全部表（步骤9）
   └── 通过 /api/write-file 写入全部表.xlsx
   → 失败：错误码 3011（error）

10. 收尾（步骤10）
    ├── 版本序列号自增并写回控制表（需加 usedRange 偏移）
    ├── 更新「表格输出」的 #工作状态# 和 #输出表格结果# 计数
    │   （仅更新工作状态和结果计数，不再写入列表和报错——已在 UI 展示）
    └── 侧边栏内联显示完整的导出结果、校验警告和错误
```

### 4.2 行版本筛选

对每一数据行（`version_r` 所在行之后的第2行起），执行以下检查：

```
输入：
  - targetVersion: 目标版本号（如 7.5）
  - primaryRoad: 版本对应的线路字段名（如 "roads_1"）
  - 当前行在版本控制区的数据：A列=版本区间, B列=roads_0值, C列=roads_1值, ...

判定流程：
  1. 读取 A 列的版本区间字符串
  2. 检查 targetVersion 是否在该区间内（调用 IsVersionInRange）
     → 不在区间内 → 排除该行
  3. 检查 roads_0 列的值：
     - 值为 "1" → 通过
     - 否则，将值作为版本区间字符串，调用 IsVersionInRange 检查
     → 不通过 → 排除该行
  4. 检查对应地区的 roads_X 列的值（同 roads_0 的逻辑）
     → 不通过 → 排除该行
  5. 三个条件全部满足 → 保留该行

关系：AND（三个条件必须同时满足）
```

**表头行（字段定义行和中文描述行）始终保留，不参与版本筛选。**

### 4.3 列版本筛选

仅当工作表中存在 `version_c` 标记时执行：

```
输入：
  - version_c 区域的数据（可能有多行：第1行=版本区间值, 后续行=线路控制值）
  - targetVersion, primaryRoad（同行筛选）

判定流程（对每一数据列）：
  1. 读取 version_c 第1行中该列对应的版本区间值
  2. 检查 targetVersion 是否在该区间内
     → 不在区间内 → 排除该列
  3. 如果 version_c 有多行（线路控制行存在）：
     - 检查 roads_0 对应的值
     - 检查 roads_X 对应的值
     → 逻辑与行筛选对称
  4. 全部条件满足 → 保留该列
```

### 4.4 版本区间解析（IsVersionInRange）

```
输入：versionRangeString（版本区间字符串）, targetVersion（目标版本号）
输出：Boolean

解析规则：
  ""（空值）     → 解析为 [0, 0.1)       → 几乎不导出
  "1.0"          → 解析为 [1.0, 99)      → 1.0及以上永久生效
  "1.0~2.5"      → 解析为 [1.0, 2.5)     → 左闭右开
  "~2.5"         → 解析为 [0, 2.5)       → 2.5以下
  "1.0~"         → 解析为 [1.0, 99)      → 同 "1.0"
  "1"            → 解析为 [1, 99)        → 值为 "1" 时视为版本区间（不是布尔开关）
  "3.5a"         → 提取数字部分 3.5      → 字母后缀（成员配置代码）被忽略
  "2s"           → 提取数字部分 2.0      → 成员代码 s 被忽略
  "1.2s~1.5s"    → 提取数字部分 [1.2, 1.5)

判定：targetVersion >= minVersion AND targetVersion < maxVersion

特殊情况：
  - roads_X 列的值为 "1" 时，含义是"在所有版本中启用"（解析为 [1, 99)）
  - roads_X 列的值为 "0" 或空时，该行/列在此线路下被排除
```

### 4.5 重复Key处理

```
筛选完成后，遍历结果数据（从第3行起，跳过两行表头）：
  - 第一数据列（#配置区域# 右侧第一列）作为 Key
  - 如果当前行的 Key 与前一行相同 → 用当前行覆盖前一行
  - 这是"简写模式"的核心机制：同一 Key 在不同版本区间有不同配置，
    筛选后保留靠后的（版本更高的）那行

示例（配置表导出版本 7.5）：
  原始数据：
    Row5: version_r=1~1.1  | id=3 | param=SUMMONS_WEAPON... | value=旧值
    Row6: version_r=1.2    | id=3 | param=SUMMONS_WEAPON... | value=新值
  筛选后（7.5 > 1.2 > 1.1，两行都保留）：
    → id=3 出现两次，保留后面那行（value=新值）
```

### 4.6 差异对比与文件写入

```
对比逻辑（ExportWriter.compareWithOldData）：
  1. 通过本地服务 /api/read-file 读取输出目录下的「全部表.xlsx」到 ExcelJS Workbook
  2. 查找英文名对应的工作表
     - 不存在 → 判定为新增，需要输出
     - 存在 → 读取旧数据，与筛选后的数据逐单元格对比
       - 行列数不同 → 判定为变更
       - 逐格 String() 比较，任何不同 → 判定为变更
  3. 特殊：英文名为 "GameConfig" 的表总是判定为变更

输出操作（仅变更的表执行）：
  1. 生成独立 .xlsx 文件（ExcelJS 生成 ArrayBuffer）
     - 文件名：{英文名}.xlsx
     - 内容：筛选后的数据（含两行表头），格式设为文本（numFmt = "@"）
     - GameConfig 特殊处理：将第3行第3列（r=2, c=2）的值替换为 "{版本号}.{序列号}"
  2. 通过本地服务 /api/write-file 将 ArrayBuffer（base64编码）写入磁盘
  3. 更新内存中的「全部表.xlsx」ExcelJS Workbook
     - 查找或创建对应英文名的工作表
     - 清空旧数据，写入筛选后的数据
  4. 所有表处理完后，将全部表 Workbook 也通过 /api/write-file 写入磁盘
```

### 4.7 导出内容的精确定义

```
导出内容 = #配置区域# 标记右侧的所有列

导出行结构（自上而下）：
  第1行：字段定义行（version_r 所在行的数据列部分）  ← 如 id=int | name=string | ...
  第2行：中文描述行（version_r 下一行的数据列部分）  ← 如 物品id | 物品名称 | ...
  第3行起：筛选后的数据行

注意：两行表头的顺序跟原表一致，不做调换。
大部分表是字段定义在前、中文描述在后，但不强制。
```

## 五、错误码机制

### 5.1 错误码定义

导出过程中任何异常都必须给出：错误码 + 出错位置 + 具体描述。策划在侧边栏看到错误后能直接定位到问题。

```typescript
interface ExportError {
  code: number;              // 错误码
  severity: 'error' | 'warning';  // error=导出中断, warning=继续但记录
  tableName: string;         // 出错的表名
  location?: {               // 出错的具体位置（可选）
    sheetName: string;
    row: number;             // 1-indexed
    column: number;          // 1-indexed
    cellValue?: string;      // 出错单元格的值
  };
  message: string;           // 人类可读的错误描述
  procedure: string;         // 出错的函数/流程名
}
```

### 5.2 错误码表

```

  1001  找不到配置设置表
  1002  找不到控制表（表格输出）
  1003  找不到表名对照表
  1010  版本序列号非数字
  1020  找不到 #版本列表# 标记
  1021  版本列表中发现重复版本名
  1030  找不到 #线路列表# 标记
  1031  线路列表中发现重复线路ID
  1040  找不到 #人员代码# 标记
  1050  找不到 #数据表版本# 标记
  1052  找不到 #输出版本# 标记
  1053  输出版本名称为空
  1054  输出版本在版本模板中未找到
  1055  找不到 #输出版本号# 标记
  1056  输出版本号非数字
  1060  找不到 #输出控制# 标记
  1061  表名对照中某行中文表名为空
  1062  表名对照中某行英文表名为空
  1070  找不到 #Git通用提交日志# 标记

2000-2099: 数据表加载错误
  2001  找不到源工作表（工作表名与表名对照中的中文名不匹配）
  2002  DataTable 对象未正确初始化
  2010  找不到 #配置区域# 标记

  2103  版本区间最小值大于最大值
  2104  版本区间波浪号两侧同时为空

2200-2299: 数据内容错误
  2201  数据区单元格值为空（位于 #配置区域# 右侧的数据区域内）

  2204  数据区域单元格仅包含空格（有内容但 trim 后为空）
  2205  配置区域单元格为空（预留，当前配置区域空值视为合法）

  2301  原始数据为空，无法执行筛选
  2302  版本控制数据格式异常

3000-3099: 导出流程错误
  3001  输出目录路径为空
  3002  输出目录不可写
  3003  创建输出目录失败
  3010  无法打开或创建「全部表.xlsx」
  3011  保存「全部表.xlsx」失败

3100-3199: 文件写入错误
  3101  独立文件写入失败（文件被占用或路径无效）
  3102  独立文件保存失败
  3103  更新全部表工作簿失败

4000-4099: Git 错误
  4001  Git 仓库路径无效
  4002  输出目录向上未找到 .git 文件夹（非 Git 仓库）
  4003  Git 命令执行失败（附带命令行输出信息）
  4004  Git pull 失败
  4005  Git push 失败
  4006  系统未安装 Git（git 命令不可用）
```

### 5.3 错误处理流程

```
导出过程中的错误分为两类：

中断性错误（error）：
  - 配置加载失败（1000系列）→ 整个导出中断
  - 输出目录不可用（3001-3003）→ 整个导出中断
  - 全部表打开/保存失败（3010-3011）→ 整个导出中断

非中断性错误（warning）：
  - 某张表加载失败（2001）→ 跳过该表，继续处理其他表
  - 某张表筛选后为空（2202）→ 跳过该表，继续
  - 某张表文件写入失败（3101-3103）→ 记录错误，继续处理其他表
  - 数据区空值（2201）→ 记录位置，继续导出

v1.0 实现：
  - 所有错误在内存中收集（ErrorHandler.errors 数组），导出完成后一次性返回给 UI
  - 侧边栏内联展示：摘要行 + 文件列表 + 警告卡片 + 错误卡片
  - 不再实时双写到「表格输出」的 #输出错误列表# 区域（列表和报错已在 UI 展示）
  - 仅更新「表格输出」的工作状态和结果计数
  - 策划可以点击警告/错误条目旁的定位按钮跳转到出错的单元格
```

### 5.4 ErrorHandler 实现（当前代码）

```typescript
class ErrorHandler {
  private errors: ExportError[] = [];

  /** 记录错误（仅写入内存，不再实时写入工作表） */
  logError(error: ExportError): void {
    this.errors.push(error);
  }

  /** 便捷方法：通过参数记录错误 */
  log(code: number, severity: ErrorSeverity, tableName: string,
      message: string, procedure: string, location?: ExportError['location']): void {
    this.logError({ code, severity, tableName, message, procedure, location });
  }

  /** 获取所有错误（供侧边栏展示） */
  getErrors(): ExportError[] {
    return [...this.errors];
  }

  getWarnings(): ExportError[] {
    return this.errors.filter(e => e.severity === 'warning');
  }

  getCriticalErrors(): ExportError[] {
    return this.errors.filter(e => e.severity === 'error');
  }

  hasCriticalErrors(): boolean {
    return this.errors.some(e => e.severity === 'error');
  }

  clear(): void {
    this.errors = [];
  }

  /**
   * 将错误列表写入「表格输出」工作表（导出完成后可选调用一次）
   * 注意：v1.0 主要通过 UI 展示错误，此方法作为备用
   */
  async writeErrorsToSheet(): Promise<void> { ... }
}
```

## 六、导出前校验

### 6.1 校验时机

在导出流程的步骤4（加载源数据到内存之后、逐表处理之前），对所有待导出的表执行一轮校验。校验不通过的项作为 warning 记录，不中断导出流程（策划可以在结果中查看）。

### 6.2 校验项（当前实现）

```
对每张待导出的数据表执行以下三类检查：

1. 配置区域校验（versionRowData）
   遍历 version_r 以下的所有数据行（索引2起）的行版本控制区域：
   - 空值允许：配置区域的空值是合法的（等同于0/不筛选）
   - 检测 Excel 错误值（#N/A, #REF! 等）→ 错误码 2203（warning）
   - 非空值执行版本格式校验：
     · 是否使用了横线 - 而非波浪号 ~  → 错误码 2101
     · 格式无效  → 错误码 1003
     · 最小值 > 最大值  → 错误码 2103
     · 波浪号两侧同时为空  → 错误码 2104
   - "0" 和 "1" 是线路开关值，跳过校验

2. 列版本区域校验（versionColData）
   与配置区域校验逻辑一致：允许空值，检测错误值，非空值检测版本格式

3. 主数据区校验（mainData）—— 检测 Ctrl+A 有效区域
   - 确定有效列数：字段定义行（row 0）中连续非空的列
   - 确定有效行数：从第3行起（索引2），遇到首列为空即停止
   - 在有效区域内检测：
     · Excel 错误值  → 错误码 2203（warning）
     · 仅包含空格的单元格（有内容但 trim 后为空）→ 错误码 2204（warning）

4. 结构完整性检查（在 DataLoader 阶段执行）
   - 找不到 version_r 标记  → 错误码 2011（warning），该表返回 null 不导出
   - 找不到 #配置区域# 标记  → 错误码 2010
   - 数据区域无列  → 错误码 2012
```

### 6.3 校验结果在侧边栏的展示

```
导出完成后，侧边栏的导出页内联显示结果，由以下几部分组成：

1. 摘要行（resultSummary）
   ✓ 导出成功  3.2s  [文件图标]5 [警告图标]3 [错误图标]0
   或
   ✗ 导出失败  1.5s  [文件图标]0 [警告图标]0 [错误图标]2

2. 文件列表卡片（resultCard）
   > SystemInfo.xlsx
   > GameConfig.xlsx
   > GameItem.xlsx
   > Equipment.xlsx
   > 全部表.xlsx

3. 警告卡片（warningCard，黄色背景 #FFF4CE）
   [dataValidation] 共 3 处警告
   [2203] 配置区域单元格包含错误值「#N/A」 (工作表: 物品表) [定位]
   [2101] 版本区间 "1.5-2.0" 使用了横线 (工作表: 装备表) [定位]
   [2204] 数据区域单元格仅包含空格 (工作表: 关卡表) [定位]
   （最多显示10条，超过显示"...等共 N 处"）

4. 错误卡片（errorCard，红色背景）
   错误 (0)  ← 仅在有错误时显示

点击 [定位] 按钮跳转到对应工作表和单元格。
```

## 七、差异对比详细逻辑

### 7.1 对比流程

```
对每张筛选完成的数据表，与「全部表.xlsx」中的对应工作表进行对比：

步骤1: 查找目标工作表
  - 在「全部表.xlsx」中查找英文名对应的工作表
  - 不存在 → 判定为「新增表」，需要输出
  - 存在 → 进入对比

步骤2: 维度对比
  - 读取旧数据的行数和列数
  - 与筛选后的行数和列数对比
  - 不同 → 判定为「变更」

步骤3: 逐单元格对比（维度相同时）
  - 将两个二维数组的每个元素转为字符串后对比
  - 任何一个不同 → 判定为「变更」
  - 全部相同 → 判定为「无变化」，跳过输出

步骤4: GameConfig 特殊处理
  - 英文名为 "GameConfig" 的表，总是判定为「变更」
  - 因为需要注入版本号，即使其他数据没变，版本号也不同

步骤5: 输出操作（仅「新增」和「变更」的表执行）
  - 写入独立 .xlsx 文件
  - 更新「全部表.xlsx」中的对应工作表
```

### 7.2 导出结果列表

```
导出完成后，变更文件列表的处理方式：

1. 侧边栏内联显示（ExportResult.modifiedFiles）
   - 每个变更表的 .xlsx 文件名（如 SystemInfo.xlsx）
   - 最后一项：全部表.xlsx
   - 在导出页的文件列表卡片中展示

2.「表格输出」工作表更新（仅更新计数，不再写入列表）
   - #工作状态# 右侧写入 "导出完成"
   - #输出表格结果# 下方写入变更文件数量
   - 不再写入 #输出表格列表# 的逐行文件路径（已在 UI 展示）
   - 不再写入 #输出错误列表#（已在 UI 展示）

注意：更新「表格输出」时需加 SheetSnapshot 的 startRow/startCol 偏移，
     因为 usedRange 可能不从 A1 开始。
```

## 八、Git 操作详细逻辑

### 8.1 v1.0 Git 策略

```
v1.0 采用「生成命令 + 用户手动执行」的方式，不自动执行 Git 命令。
原因：Office Web Add-in 运行在浏览器沙箱中，无法直接执行系统命令。

工作流程：
  1. 用户在导出页点击「开始导出」→ 完成数据筛选和文件写入
  2. 用户点击导出页的「Git」按钮 → 生成完整推送脚本并复制到剪贴板
  3. 用户在终端中粘贴执行

未来版本（v2.0+）可通过 companion server 实现自动执行。
```

### 8.2 Git 命令生成

```
GitHandler 类负责生成命令字符串，不执行任何实际 Git 操作。

拉取命令（generatePullCommands）：
  cd "{输出目录}" && git reset --hard && git clean -dfq && git pull

推送命令（generatePushCommands）：
  cd "{输出目录}" && git add "file1.xlsx" && git add "file2.xlsx" && ...
  && git commit -m "{提交信息}" && git push

提交信息生成（generateCommitMessage）：
  模板从配置设置表 #Git通用提交日志# 读取
  支持占位符：{VerName} {VerNum} {SeqNum} {User} {Date}
  拼接方式：模板替换后 + 空格 + 版本名 + 空格 + 版本号 + "." + 序列号

getFullPushScript() 将所有命令用 " && " 连接为单行脚本。
```

### 8.3 导出页的 Git 按钮

```
位置：导出页操作区，与「开始导出」按钮并排

点击行为：
  1. 生成提交信息（使用 GitHandler.generateCommitMessage）
  2. 生成完整推送脚本（使用 GitHandler.getFullPushScript）
  3. 复制到剪贴板（navigator.clipboard.writeText，降级使用 execCommand）
  4. 弹出提示：「Git 推送命令已复制到剪贴板，请在终端中粘贴执行。」

启用条件（canGitPush）：
  - 不在导出中（!isExporting）
  - 导出成功（exportResult.success）
  - 有变更文件（modifiedFiles.length > 0）
```

### 8.4 Git 页签

```
Git 页签为简化的设置界面（GitPanel 组件），展示以下信息：

1. 仓库目录
   显示 config.outputSettings.outputDirectory

2. 提交信息预览
   显示根据当前配置生成的提交信息

3. 底部提示
   "v1.0：导出后点击「Git」按钮可复制推送命令到剪贴板。"

不显示：推送命令详情、变更文件列表、Git 环境检测状态
这些信息已在导出页内联展示或通过 Git 按钮操作。
```

## 九、配置设置表中的版本-线路映射详解

```
版本选择流程：

  1. 用户在「表格输出」的 #输出版本# 位置选择版本名（如 "国内"）
  2. 在「配置设置表」的 #版本列表# 中查找 "国内"
     → 获取输出线路编号 = 2
     → 获取 Git 目录模板 = C:\...\{0}
  3. 在「配置设置表」的 #线路列表# 中查找线路 ID = 2
     → 获取线路字段名 = roads_1
  4. 导出时对每行数据检查：
     - roads_0（总线路，固定检查）—— AND
     - roads_1（国内线路）—— 都通过才保留

简化规则：一个版本 = 一条线路 = 一个地区（无多线路支持）
  主干版本 → roads_0（总线路，lineField = roads_0 时只检查 roads_0）
  国内     → roads_0 + roads_1
  版号包   → roads_0 + roads_2
  日本     → roads_0 + roads_4（注意不是 roads_3）
  韩国     → roads_0 + roads_4（→实际应为 roads_3 + roads_0，需按配置表实际数据）
  ...

⚠ roads_0 是总开关，所有版本导出时都会检查 roads_0。
  如果某行 roads_0 = 0，则该行在所有版本中都不会导出。

⚠ 当 targetLineField === 'roads_0' 时，只检查 roads_0，不重复检查。
  （代码中 DataFilter.filterRows 有 targetLineField !== 'roads_0' 的判断）
```

## 十三、VBA 已知 Bug 及 Add-in 修正建议

在分析 VBA 代码过程中发现以下问题，Add-in 实现时应修正：

1. **clsDataTable.ApplyFilters 列筛选中的变量名错误**  
   VBA 代码第二遍列筛选中，`includeRow = False` 应为 `includeCol = False`。  
   → Add-in 中使用独立的 `includeCol` 变量。

2. **modErrorHandler.InitializeFallbackErrorLog 中错误使用 New**  
   `Dim modUtilities As modUtilities` 然后 `Set modUtilities = New modUtilities`，标准模块不能 New。  
   → Add-in 中 utils 模块使用纯函数导出，不存在此问题。

3. **LogToFile 默认禁用**  
   `LogToFile` 函数入口直接 `Exit Sub`，调试日志实际不工作。  
   → Add-in 中实现可开关的 console + 文件双写日志。

## 十五、已确认事项与后续计划

### 已解决

1. **文件系统访问**：通过本地开发服务器的 `/api/write-file` 和 `/api/read-file` 端点实现。ExportJob 将 ExcelJS 生成的 ArrayBuffer 转为 base64 后发送到服务端写入磁盘。

2. **Git 命令执行**：v1.0 采用生成命令 + 复制到剪贴板的方案。用户在终端粘贴执行。未来版本可接入 companion server 自动执行。

3. **version_c 多行线路控制**：DataLoader 已实现 versionColLabels 提取，DataFilter 使用 colLabels 识别 roads_0 和目标线路行。支持 version_c 区域有多行的情况。

4. **无 version_r 的表处理**：DataLoader 中找不到 version_r 标记时返回 null，该表不参与导出。例如技能buff表等辅助表。

### 后续计划

1. **网页协同导出**：v1.0 暂未实现 OneDrive 同步方式的协同导出，后续版本考虑。

2. **预览功能**：导出页的「预览」tab 目前显示"开发中"占位符。

3. **独立校验功能**：「校验」tab 目前显示"开发中"占位符，校验逻辑已集成在导出流程中。

4. **usedRange 偏移**：SheetSnapshot 包含 startRow/startCol，所有 getRangeByIndexes 调用需加偏移。这是关键的正确性要求，在 ConfigLoader.incrementSequence 和 ExportJob.updateExportResults 中已正确处理。
