## 一、v2.0 架构概览

v2.0 的核心变化是引入 **StudioConfig JSON 存储**，将所有配置集中到 `StudioConfig` 工作表 A1 单元格的一个 JSON 字符串中，取代旧版分散在多张工作表中的标记格式。旧格式（「表格输出」「配置设置表」「表名对照」）仅作为回退兼容。

### 1.1 模块总览

| 模块 | 文件路径 | 职责 |
|------|---------|------|
| StudioConfigStore | `src/v2/StudioConfigStore.ts` | JSON 配置的读写核心，`StudioConfig` A1 存储 |
| ConfigManager | `src/v2/ConfigManager.ts` | 配置项的增删改（版本/线路/人员/开关等） |
| TableRegistry | `src/v2/TableRegistry.ts` | 数据表注册、扫描、同步到表名对照 |

## 二、StudioConfigStore — JSON 存储核心

### 2.1 存储方式

`StudioConfig` 工作表（常量 `SHEET_CONFIG = 'StudioConfig'`，定义于 `TemplateFactory.ts`）的 **A1 单元格**存储完整的 `StudioConfigData` JSON 字符串。工作表创建后默认设为 `hidden`。

### 2.2 StudioConfigData 结构

```typescript
interface StudioConfigData {
  outputVersion: string;         // 输出版本名
  outputVersionNumber: number;   // 输出版本号
  versionSequence: number;       // 版本序列号
  fullVersion: string;           // 完整版本号字符串 (如 "7.5.1152")
  workStatus: string;            // 工作状态
  resultCount: number;           // 输出表格结果数量

  versions: VersionTemplate[];   // 版本列表
  lines: LineTemplate[];         // 线路列表
  staff: StaffInfo[];            // 人员代码

  gitCommitTemplate: string;     // Git 提交模板
  switches: Record<string, boolean>;  // 功能开关
  tables: TableInfo[];           // 输出控制（表名对照）

  tableSchema?: TableSchemaDoc;  // 数据表格式说明文档（纯文档，不参与业务逻辑）
}
```

默认配置通过 `createDefaultConfig()` 生成：
- `outputVersion: '默认'`，`outputVersionNumber: 1`
- 默认包含一个版本 `{ name: '默认', lineId: 0, lineField: 'roads_0', gitDirectory: '' }`
- 默认包含一条线路 `{ id: 0, field: 'roads_0', remark: '默认线路' }`
- 默认包含一个人员 `{ id: 1, name: '默认用户', code: 'default', machineCode: '' }`
- 默认开关 `{ '自动弹出路径': false }`
- `tables: []`（空），创建时自动附带 `tableSchema`

### 2.3 核心 API

```typescript
class StudioConfigStore {
  // 从 StudioConfig A1 读取 JSON，不存在返回 null
  static async load(context: Excel.RequestContext): Promise<StudioConfigData | null>;

  // 序列化写入 StudioConfig A1（每次保存自动刷新 tableSchema）
  static async save(context: Excel.RequestContext, data: StudioConfigData): Promise<void>;

  // 创建 StudioConfig 工作表 + 写入默认/指定配置，visibility 设为 hidden
  static async create(context: Excel.RequestContext, data?: StudioConfigData): Promise<void>;

  // 便捷方法：load → mutator(data) → save，不存在返回 false
  static async update(
    context: Excel.RequestContext,
    mutator: (data: StudioConfigData) => void
  ): Promise<boolean>;
}
```

行为细节：
- `load` 时自动补全 `tableSchema`（若已有配置但缺少此字段，则补写并立即回写 A1）
- `save` 时每次用 `createTableSchema()` 覆盖 `tableSchema`，保证文档始终为最新版本
- `create` 在工作表已存在时抛出错误

### 2.4 buildRoadsFromConfig

统一的 roads 列表构建方法，所有需要 roads 列表的地方都应调用此方法：

```typescript
function buildRoadsFromConfig(
  data: StudioConfigData
): Array<{ field: string; name: string }>
```

逻辑：
1. 构建 `lineId → field` 映射（来自 `data.lines`，是真实来源）
2. `roads_0`（名称「默认」）始终作为第一个元素存在
3. 遍历 `data.versions`，优先用 `lines` 中的 field，其次用 version 自带的 `lineField`
4. 跳过 `roads_0` 和非 `roads_` 前缀的字段
5. 按 roads 编号（数字部分）升序排序

### 2.5 tableSchema — 表格格式说明文档

`StudioConfigData.tableSchema` 是纯文档性质的字段，描述数据表工作表的完整布局规范。AI 或外部工具可据此自动生成符合规范的数据表。由 `createTableSchema()` 生成，包含以下子文档：

| 子文档 | 描述 |
|--------|------|
| `sheetLayout` | 工作表布局：versionC 区域、versionR 行、描述行、数据行、#配置区域# 标记 |
| `fieldFormat` | 字段定义语法 `[prefix_]fieldName=type`，前缀（key_/language_）和类型（int/float/string/数组） |
| `versionRange` | 版本区间规则（左闭右开 `[min, max)`） |
| `roadsControl` | 线路控制列的筛选逻辑（roads_0 总开关 + roads_N 地区专属） |
| `examples` | 仅行控制模式（R）和行列双控模式（R+C）的完整示例网格 |

## 三、ConfigManager — 配置管理

### 3.1 设计模式

所有写入操作统一采用 **JSON 优先 + 旧格式回退**：

```
StudioConfigStore.update() → 成功 → 结束
                           → 返回 false → 回退到旧格式标记工作表写入
```

### 3.2 支持的操作

```typescript
class ConfigManager {
  // 版本模板（操作 data.versions + 旧格式 #版本列表#）
  async addVersion(version: VersionTemplate): Promise<void>;
  async updateVersion(oldName: string, updated: VersionTemplate): Promise<void>;
  async deleteVersion(name: string): Promise<void>;
  // deleteVersion 同时删除对应的 lines 记录（通过 lineId 关联）

  // 线路模板（操作 data.lines + 旧格式 #线路列表#）
  async addLine(line: LineTemplate): Promise<void>;
  async updateLine(id: number, updated: LineTemplate): Promise<void>;

  // 人员代码（操作 data.staff + 旧格式 #人员代码#）
  async addStaff(staff: StaffInfo): Promise<void>;
  async updateStaff(name: string, updated: StaffInfo): Promise<void>;

  // Git 提交模板
  async setGitCommitTemplate(template: string): Promise<void>;

  // 功能开关
  async setSwitch(name: string, value: boolean): Promise<void>;

  // 输出设置
  async setOutputVersion(versionName: string): Promise<void>;
  async setOutputVersionNumber(versionNumber: number): Promise<void>;
}

// 单例导出
export const configManager = new ConfigManager();
```

### 3.3 旧格式兼容辅助

通过 `loadLegacySnap(context, marker, legacySheet)` 加载旧格式工作表快照，使用 `excelHelper.findMarkerInData()` 查找标记位置，`excelHelper.readBlockBelow()` 读取标记下方数据块。

旧格式工作表名常量：
- `LEGACY_CONTROL = '表格输出'`
- `LEGACY_SETTINGS = '配置设置表'`

输出设置的旧格式写入通过 `writeLegacyMarkerRight()` 私有方法完成（标记右侧单元格写入值）。

## 四、TableRegistry — 数据表注册管理

### 4.1 核心功能

```typescript
interface UnregisteredTable {
  sheetName: string;
  hasConfigMarker: boolean;  // 是否包含 #配置区域#（实际只返回有标记的表）
}

class TableRegistry {
  // 扫描未注册的数据表（排除系统表，仅返回包含 #配置区域# 标记的工作表）
  async scanUnregistered(): Promise<UnregisteredTable[]>;

  // 注册新表（写入 JSON + syncToLegacyMapping）
  async registerTable(info: TableInfo): Promise<void>;

  // 修改已注册表信息
  async updateTable(chineseName: string, updates: Partial<TableInfo>): Promise<void>;

  // 取消注册（可选同时删除工作表，默认 deleteSheet = false）
  async unregisterTable(chineseName: string, deleteSheet?: boolean): Promise<void>;

  // 获取所有已注册表（优先 JSON，回退旧格式）
  async getRegisteredTables(): Promise<Map<string, TableInfo>>;
}

export const tableRegistry = new TableRegistry();
```

### 4.2 系统表排除列表

以下工作表不参与扫描：

```typescript
const SYSTEM_SHEETS = new Set([
  '表格输出', '配置设置表', '表名对照', '模板说明', '说明表', 'StudioConfig'
]);
```

### 4.3 syncToLegacyMapping — 表名对照同步

每次 JSON 配置中的 `tables` 更新后（注册/修改/取消注册），自动全量同步到「表名对照」工作表的 `#输出控制#` 区域：

1. 读取当前 `StudioConfigData.tables`
2. 加载「表名对照」快照，定位 `#输出控制#` 标记
3. 清除标记下方所有旧数据行
4. 写入最新的 `[versionRange, chineseName, englishName, shouldOutput]` 行

这确保了「表名对照」始终与 JSON 配置保持一致（给玩家或其他工具查看）。

注意：`registerTable` 和 `updateTable` 在 JSON 更新成功后均调用 `syncToLegacyMapping`。`unregisterTable` 仅在 JSON 更新成功时同步（旧格式回退路径直接操作表名对照）。

### 4.4 删除表二次确认

在 UI 层（TablesSubPage），删除操作实现了内联二次确认：

1. 点击删除按钮 → 显示确认（CheckmarkRegular）/取消（DismissRegular）两个小按钮
2. 确认后调用 `tableRegistry.unregisterTable(chineseName, true)` — 同时删除工作表 + 取消注册
3. 取消后恢复原始删除按钮

## 五、TableCreator — 新建数据表

### 5.1 创建流程

```typescript
interface FieldDefinition {
  name: string;           // 如 "id"
  type: string;           // 如 "int"
  description: string;    // 如 "怪物ID"
  isKey: boolean;         // 加 key_ 前缀
  isLanguage: boolean;    // 加 language_ 前缀
}

interface TableCreationConfig {
  chineseName: string;        // 工作表名（中文）
  englishName: string;        // 导出文件名（英文）
  startVersion: string;       // 起始版本号
  fields: FieldDefinition[];  // 字段定义列表
  includeVersionCol: boolean; // 是否包含 version_c 列控制
  autoRegister: boolean;      // 是否自动注册到表名对照
}
```

注意：`FieldDefinition` 和 `TableCreationConfig` 同时定义在 `src/v2/TableCreator.ts` 和 `src/types/studio.ts` 中，ManageTab 从 `TableCreator.ts` 导入。

### 5.2 整行批量写入

TableCreator 的关键实现特点是**整行批量写入**，避免逐单元格写入可能导致的数据丢失。

列布局计算：
```
configMarkerCol = 1 + roadsCount + 2(空隙列)
dataStartCol = configMarkerCol + 1
totalCols = dataStartCol + fields.length
```

**version_c 行**（仅当 `includeVersionCol = true` 时写入第 0 行）：
- `configMarkerCol - 1` 列写入 `'版本列属'`
- `configMarkerCol` 列写入 `'version_c'`
- 数据列区域写入 `config.startVersion`（所有数据列初始版本相同）

**version_r 行**（`vrRow = 0` 或 `vrRow = 4`，取决于是否有 version_c）：
```typescript
vrRowData[0] = 'version_r';
vrRowData[1..roadsCount] = roads[i].field;  // roads_0, roads_1, ...
vrRowData[configMarkerCol] = '#配置区域#';
vrRowData[dataStartCol..] = fieldStrs;      // 如 'key_id=int', 'name=string'
```

**描述行**（`vrRow + 1`）：
- A 列写入 `'版本行属'`
- roads 区域写入各线路中文名
- 数据区域写入各字段的中文描述

创建完成后调用 `sheet.activate()` 激活新工作表。

### 5.3 Roads 来源

通过 `StudioConfigStore.load()` + `buildRoadsFromConfig()` 获取 roads 列表，仅包含当前配置中已有的版本（而非全部线路 ID）。若 StudioConfig 不存在，退化为 `[{ field: 'roads_0', name: '默认' }]`。

### 5.4 自动注册

当 `autoRegister = true` 时，创建完工作表后（在 `context.sync()` 之后的独立步骤中）通过 `tableRegistry.registerTable()` 注册到 JSON 配置和表名对照。

### 5.5 撤销

`undoLastCreation()` 删除最近创建的工作表，若当时自动注册了还会通过 `tableRegistry.unregisterTable()` 取消注册。仅保留最近一次创建记录。

### 5.6 字段名构建

`buildFieldString()` 将 `FieldDefinition` 转换为字段定义字符串：
- `isKey = true` → `key_{name}={type}`
- `isLanguage = true` → `language_{name}={type}`
- `isKey` 优先于 `isLanguage`（两者互斥，`isKey` 优先检查）

## 六、LineSyncer — 线路同步三阶段

### 6.1 入口

```typescript
class LineSyncer {
  async syncAllTables(
    versionTemplates: Map<string, VersionTemplate>,
    tableSheetNames: string[]
  ): Promise<{ synced: number; skipped: number; errors: string[] }>;
}

export const lineSyncer = new LineSyncer();
```

入口方法先从 `versionTemplates` 中收集所有需要的 roads（`roads_0` 始终包含），按编号排序，并构建 `roadsToName` 映射（`roads_0` → `'默认'`，其余取 version 的 name）。然后逐表调用 `syncSingleTable`。

### 6.2 三阶段详解

对每张表在单个 `Excel.run` 上下文中执行以下三个阶段，每个阶段完成后 **reload 快照**确保位置信息准确：

#### 阶段一：version_r 列同步（整列增删）

1. 加载工作表快照，定位 `version_r` 行
2. 收集 version_r 行中现有的 roads 列（field → colIndex），遇到空列或 `#配置区域#` 时停止
3. **删除多余列**：从右往左整列删除，避免索引偏移
4. **添加缺失列**：在最后一个 roads 列后批量插入整列（`insert(Excel.InsertShiftDirection.right)`），写入：
   - version_r 行：roads 字段名
   - 描述行：版本中文名
   - 数据行：复制 roads_0 列的全部数据作为默认值

#### 阶段二：version_c 行同步（整行增删，仅 R+C 模式）

仅当工作表包含 `version_c` 标记且在 version_c 与 version_r 之间存在 roads 行时执行：

1. 收集 version_c 和 version_r 之间的 roads 行（field → rowIndex）
2. **删除多余行**：从下往上整行删除
3. **添加缺失行**：在最后一个 roads 行后插入整行（`insert(Excel.InsertShiftDirection.down)`），写入：
   - roads 字段名（version_c 同列）
   - 版本名标签（version_c 左一列）
   - 数据列默认值全部设为 1
4. 数据列数从 version_r 行的 `#配置区域#` 后统计（可靠来源，不受 vc 表头污染）

#### 阶段三：版本名同步

1. **version_r 描述行**：对比 roads 列对应的中文名，不一致则更新
2. **version_c 标签列**：对比 roads 行左一列（`vcPos.col - 1`）的中文名，不一致则更新

### 6.3 标记查找

`findMarker()` 在前 30 行内搜索标记文字：
- `version_r`：限定前 5 列搜索（因为 version_r 一定在前几列）
- `version_c`：不限列数（version_c 可能在较远的列，即 `#配置区域#` 附近）

## 七、TemplateFactory — 模版创建与迁移

### 7.1 一键创建

```typescript

  async createTemplate(): Promise<void>;

}

```

### 7.2 旧格式迁移

`migrateFromLegacy()` 读取旧格式三张表，构建完整的 `StudioConfigData`：

| 旧格式来源 | JSON 字段 |
|-----------|----------|
| 表格输出 → `#输出版本#` 右侧 | `outputVersion` |
| 表格输出 → `#输出版本号#` 右侧 | `outputVersionNumber` |
| 表格输出 → `#数据表版本#` 区域（同行右侧 + 上一行右侧） | `fullVersion` / `versionSequence` |
| 表格输出 → `#工作状态#` 右侧 | `workStatus` |
| 配置设置表 → `#版本列表#` 下方（4列） | `versions[]` |
| 配置设置表 → `#线路列表#` 下方（3列） | `lines[]` |
| 配置设置表 → `#人员代码#` 下方（4列） | `staff[]` |
| 配置设置表 → `#Git通用提交日志#` 下方（1列） | `gitCommitTemplate` |
| 配置设置表 → `#配置开关#` 下方（2列） | `switches{}` |
| 表名对照 → `#输出控制#` 下方（4列） | `tables[]` |

迁移时先以 `createDefaultConfig()` 为基础，逐步覆盖各字段。迁移完成后调用 `StudioConfigStore.create(context, data)` 写入。若 StudioConfig 已存在则跳过。

### 10.3 ManageTab — 管理面板

ManageTab 包含三个子面板，通过顶部子导航切换：

```
┌────────────────────────────┐
│  [配置]  [表管理]  [新建表] │
├────────────────────────────┤
│  子面板内容                 │
└────────────────────────────┘
```

子面板类型：`type ManageSubPage = 'config' | 'tables' | 'wizard'`

#### ConfigSubPage — 配置管理

| 区域 | 功能 |
|------|------|
| 操作员 | 下拉选择（来自人员代码列表），通过 `operatorIdentity.set()` 持久化 |
| 版本管理 | 表格展示版本列表（版本名/线路/Git 目录），支持行内编辑 Git 目录、添加/删除版本、同步线路 |
| 人员代码 | 表格展示人员列表（姓名/代码），支持行内编辑（点击文字进入编辑模式）、添加 |
| Git 提交模板 | 显示/编辑提交模板（Textarea），保存到 `configManager.setGitCommitTemplate()` |
| 功能开关 | Switch 组件切换（当前支持「自动弹出路径」） |
| StudioConfig | 显示/隐藏 StudioConfig 工作表的 Switch + 一键创建模版按钮 |

**添加版本流程**：
1. Git 目录为必填项（无输出目录的线路没有意义）
2. 自动计算下一个线路编号（取现有最大 roads 编号 + 1）
3. 同时调用 `configManager.addVersion()` 和 `configManager.addLine()` 保持一致
4. 添加后提示用户点击「同步线路」将新列同步到所有数据表

**删除版本**：`roads_0`（默认线路）不可删除（按钮不显示）。删除调用 `configManager.deleteVersion()`，内部同时删除对应的 lines 记录。

**同步线路**：调用 `lineSyncer.syncAllTables()`，传入版本模板和所有已注册数据表名称。

#### TablesSubPage — 表管理

| 功能 | 说明 |
|------|------|
| 扫描 | 调用 `tableRegistry.scanUnregistered()` 扫描未注册表 |
| 未注册表列表 | 显示带「有配置」标签的未注册表，一键注册（英文名默认与中文名相同） |
| 搜索 | 按中文名/英文名过滤 |
| 已注册表列表 | 表格展示（中文名/英文名/版本/控制/删除） |
| 版本号编辑 | 行内点击编辑版本区间，调用 `tableRegistry.updateTable()` |
| 版本控制三态切换 | 点击 badge 循环切换 R → R+c → R+C → R |
| 删除 | 内联二次确认（确认/取消按钮），调用 `tableRegistry.unregisterTable(name, true)` |

**版本控制三态检测**：

页面加载时自动检测所有已注册表的版本控制状态：
- 在每张表的前 30 行内搜索 `version_c` 和 `version_r`
- 若无 `version_c` 或 `version_r` 在 `version_c` 上方 → `R`
- 若有 `version_c`，检查其与 `version_r` 之间是否有 `roads_` 行 → 有则 `R+C`，无则 `R+c`

**版本控制三态**：

| 状态 | 含义 | 工作表结构 |
|------|------|-----------|
| R | 仅行控制 | 无 version_c 区域 |
| R+c | 行 + 简单列控制 | version_c 行 + 2 空行（无 roads 行） |
| R+C | 行 + 完整列控制 | version_c 行 + roads 行 + 空行 |

切换逻辑（每次切换前先同步该表的线路列）：
- **R → R+c**：在 version_r 上方插入 3 行（1 行 version_c + 2 行空行），写入标记、`'版本列属'` 和数据列默认值 1
- **R+c → R+C**：在 version_c 下方插入 roads 行，写入字段名、版本名和默认值 1
- **R+C → R**：删除 version_c 到 version_r 之间的所有行（`vrRow - vcRow` 行）

version_c 的位置约定：`version_c` 与 `#配置区域#` 同列，`'版本列属'` 在其左一列。

#### WizardSubPage — 新建表

新建表向导表单，使用 `TableCreator` 创建工作表。

表单字段：
- 中文名（工作表名）
- 英文名（导出文件名）
- 起始版本号（默认 "1"）
- 是否包含列控制（version_c）
- 是否自动注册（默认 true）
- 字段列表（可动态添加/删除，每个字段包含名称、类型、描述、isKey、isLanguage）

默认预填一个字段：`{ name: 'id', type: 'int', description: 'ID', isKey: true, isLanguage: false }`。

`TableCreator` 实例化为模块级变量（`const tableCreator = new TableCreator()`），在 WizardSubPage 中共享。

## 十二、与旧格式的兼容策略

### 12.1 读取优先级

所有模块统一遵循：

```
优先读取 StudioConfig (JSON)
  ↓ 不存在
回退读取旧格式工作表（表格输出/配置设置表/表名对照）
```

### 12.2 写入策略

- **ConfigManager**：JSON 优先写入。若 `StudioConfigStore.update()` 返回 false（StudioConfig 不存在），回退到旧格式标记工作表写入。
- **TableRegistry**：JSON 更新成功后，额外调用 `syncToLegacyMapping()` 全量同步到「表名对照」工作表，确保旧格式数据与 JSON 一致。
- **TemplateFactory**：提供 `migrateFromLegacy()` 从旧格式一键迁移到 JSON。

### 12.3 迁移路径

1. 旧工作簿首次使用 v2.0：所有操作通过旧格式回退正常工作
2. 用户在管理面板点击「一键创建模版」或手动迁移：`templateFactory.createTemplate()` 或 `templateFactory.migrateFromLegacy()`
3. 迁移后所有操作优先通过 JSON，旧格式表名对照保持同步
